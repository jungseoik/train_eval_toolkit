## 1. 전체 흐름

```
CLI (main.py autolabel)
  → API 연결 사전 검증
  → 파일 수집 (os.walk 재귀 탐색)
  → YAML에서 프롬프트 룩업
  → ProcessPoolExecutor 병렬 분배
  → 각 워커: API 호출 → JSON 파싱 → description 검증 (최대 3회 재시도)
  → 원본 파일 옆에 .json 저장
  → 실패 목록 → assets/logs/에 기록
```

---

## 2. API 연결 사전 검증

병렬 처리를 시작하기 전에 `GeminiClient.validate()`로 1회 검증 요청을 전송합니다.

| 항목 | 내용 |
|------|------|
| 목적 | 복구 불가능한 에러를 초기 단계에서 감지 |
| 감지 대상 | API 키 오류, 모델명 오류, 네트워크 오류 |
| 실패 시 동작 | 즉시 중단 (파일 처리 시작하지 않음) |
| 성공 시 동작 | 검증 성공 메시지 출력 후 본 처리 시작 |

---

## 3. 재시도 로직

각 파일에 대해 최대 **3회** 시도를 수행합니다.

### 재시도 대상 조건

- API 빈 응답
- API 호출 예외 (일시적 네트워크 오류, rate limit 등)
- JSON 파싱 실패 (LLM이 깨진 텍스트 응답)
- `description` 키 누락 또는 비어있음

### 재시도 정책

- **재시도 간격**: 1초
- **최종 실패 처리**: 3회 전부 실패 시 해당 파일은 `None` 반환 후 실패 로그에 기록

---

## 4. JSON 파싱 전략

LLM 응답의 형식이 일정하지 않을 수 있으므로 **2단계 파싱**을 적용합니다.

### 파싱 단계

1. **1단계**: 마크다운 코드 블록(` ```json {...} ``` `) 패턴 추출
2. **2단계**: 텍스트에서 첫 번째 `{` ~ 마지막 `}` 범위를 추출하여 `json.loads()` 수행

### 키 처리 방침

- 키 제한 없음: JSON 내 모든 키가 그대로 보존됩니다.
- `category`, `description` 외 추가 키도 결과 파일에 저장됩니다.

### 파싱 후 검증

파싱 완료 후 `description` 키가 존재하고 비어있지 않은지를 추가로 검증합니다.

## 5. 에러 처리 구조

각 레이어가 에러를 명확하게 분리하여 처리합니다.

| 레이어 | 위치 | 처리 방식 |
|--------|------|-----------|
| API 호출 | `client.py` — `GeminiClient` | 호출 실패 시 예외를 그대로 전파 (에러 문자열 반환하지 않음) |
| 단일 파일 처리 | `labeler.py` — `_label_single_file` | `try-except`로 예외를 잡아 재시도 루프에서 처리 |
| 전체 처리 진입 | `labeler.py` — `autolabel_files_recursively` | 사전 검증 실패 시 `return`으로 즉시 종료 |

---

## 6. 실패 로그

처리에 실패한 파일 목록은 별도의 로그 파일에 기록됩니다.

| 항목 | 내용 |
|------|------|
| 저장 경로 | `assets/logs/failed_files_{timestamp}.txt` |
| 기록 형식 | 실패한 파일의 절대 경로를 한 줄에 하나씩 기록 |
| 터미널 출력 | 실패 수와 로그 파일 경로를 함께 출력 |
| 완료 요약 | 전체 처리 완료 시 성공 수 / 전체 수 요약 출력 |

---

## 7. 덮어쓰기 동작 (`--overwrite`)

`--overwrite` 옵션은 기존 라벨 파일의 처리 여부를 제어합니다.

| 모드 | 동작 |
|------|------|
| 기본값 (`False`) | 기존 `.json`이 있고 `description`이 정상이면 스킵 |
| `--overwrite` 사용 | 기존 파일을 무시하고 무조건 재라벨링 |
| 예외 | `description`이 비어있거나 JSON이 깨진 경우는 `--overwrite` 없이도 재처리 |

---

## 8. 스킵 판정 기준

기존 JSON 파일이 존재하고 `overwrite=False`인 경우에 한해 아래 기준으로 스킵 여부를 판정합니다.

| 조건 | 판정 |
|------|------|
| `description` 키 존재 + 값이 비어있지 않음 | 스킵 |
| `description`이 비어있음 | 재처리 |
| JSON 파싱 실패 (깨진 파일) | 재처리 |
